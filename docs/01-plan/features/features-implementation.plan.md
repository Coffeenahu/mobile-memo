## 1. 메모 색상 라벨 (Color Label)

### 1.1 목적
메모 카드에 색상 라벨을 추가하여 시각적으로 메모를 구분하고, 사용자가 메모를 빠르게 식별할 수 있도록 한다.

### 1.2 구현 개요
- **UI**: 메모 카드 좌측에 색상 바 표시, 목록 뷰에 색상 필터 버튼
- **상태**: 각 메모에 color 필드 추가, 선택된 필터 색상 상태 관리
- **로직**: 메모 생성/수정 시 색상 선택, 색상별 필터링, 기본값 제공
- **데이터**: LocalStorage에 color 필드 및 필터 상태 포함

### 1.3 구현 단계
#### 단계 1: 데이터 모델 확장
- `{ id, title, body, createdAt, updatedAt, starred, color }` 구조로 확장
- 기존 메모에는 기본 색상('gray') 적용

#### 단계 2: HTML 변경 (index.html)
- 메모 편집/생성 뷰에 색상 선택 UI(라디오 버튼 또는 select) 추가
- 메모 카드에 색상 바(div) 추가

#### 단계 3: CSS 스타일 추가 (css/style.css)
- 색상 바 스타일 정의 (예: `.note-label--red { background: #ff6b6b; }` 등)

#### 단계 4: JavaScript 구현 (js/app.js)
- 메모 생성/수정 시 color 필드 저장
- 렌더링 시 색상 바에 맞는 클래스 적용
- 기존 메모 마이그레이션 처리(초기화 시 color 없으면 'gray' 할당)

#### 단계 5: 색상 필터 UI 추가 (index.html)
- 목록 뷰에 색상 필터 버튼 그룹 추가
- 각 색상별 토글 버튼 (gray, red, blue, green, yellow)

#### 단계 6: 필터링 로직 구현 (js/app.js)
- 선택된 색상 상태 관리 (state.activeColorFilter)
- 필터링 함수: 선택된 색상만 표시, 'all' 선택 시 전체 표시
- 필터 상태 LocalStorage에 저장/복원

### 1.4 테스트 계획
- 색상 선택/저장/렌더링 정상 동작 확인
- 기존 메모도 문제없이 표시되는지 확인
- 색상별 시인성, 접근성 확인
- 색상 필터 선택 시 해당 색상 메모만 표시되는지 확인
- '전체' 필터 선택 시 모든 메모 표시되는지 확인
- 필터 상태가 앱 재시작 후 유지되는지 확인

### 1.5 완료 기준
- [ ] 색상 선택 UI 구현
- [ ] 메모 카드에 색상 바 표시
- [ ] 데이터 구조 확장 및 마이그레이션
- [ ] 색상 필터 UI 및 로직 구현
- [ ] 필터 상태 저장/복원
- [ ] 모든 테스트 케이스 통과

### 1.6 예상 소요 시간
- 구현: 1.5시간
- 테스트: 30분

## 2. 다크모드 (Dark Mode)

### 2.1 목적
다크모드를 추가하여 야간 사용성을 향상시킨다. 시스템 설정(prefers-color-scheme)에 자동으로 연동된다.

### 2.2 구현 개요
- **UI**: 변경 없음 (자동 적용)
- **상태**: 추가 상태 불필요
- **로직**: CSS 변수로 색상 관리, prefers-color-scheme 자동 연동
- **데이터**: 추가 필드 불필요, CSS 클래스 토글

### 2.3 구현 단계
#### 단계 1: CSS 변수 및 다크모드 스타일
기존 색상을 CSS 변수로 변환하고 prefers-color-scheme 미디어 쿼리로 다크모드 자동 적용.

#### 단계 2: HTML 변경
없음 - 자동 적용.

#### 단계 3: JavaScript 변경
테마 적용 함수 추가 및 초기화 시 호출.

### 2.4 테스트 계획
#### 단위 테스트 (수동)
1. 시스템이 다크모드일 때 자동 적용되는지 확인
2. 시스템 설정 변경 시 즉시 반영되는지 확인
3. 밝은 테마/어두운 테마 모두 올바르게 표시되는지 확인

#### 엣지 케이스
- prefers-color-scheme 미지원 브라우저 (기본 밝은 테마 유지)
- 시스템 설정 변경 중 앱 사용

### 2.5 완료 기준
- [ ] CSS 변수로 색상 체계 구축
- [ ] prefers-color-scheme 미디어 쿼리 적용
- [ ] 시스템 설정 변경 시 자동 반영 확인
- [ ] 모든 테스트 케이스 통과

### 2.6 예상 소요 시간
- 구현: 30분-1시간
- 테스트: 15분

### 2.7 다음 단계
구현 완료 후 PDCA 사이클의 Do 단계로 이동하여 실제 코딩 시작.

## 3. 내보내기/가져오기 (Export/Import)

### 3.1 목적
메모 데이터를 JSON 파일로 내보내고, 다시 가져올 수 있도록 하여 데이터 백업·복원 및 기기 이전을 지원한다.

### 3.2 구현 개요
- **UI**: 목록 뷰에 내보내기(Export)·가져오기(Import) 버튼 추가
- **내보내기**: 현재 LocalStorage의 모든 메모를 JSON 파일로 다운로드
- **가져오기**: JSON 파일을 선택해 기존 메모에 병합 또는 덮어쓰기
- **데이터**: 기존 메모 데이터 구조 유지

### 3.3 구현 단계
#### 단계 1: HTML 변경 (index.html)
목록 뷰 헤더 또는 하단에 버튼 추가:

```html
<button id="btn-export">내보내기</button>
<button id="btn-import">가져오기</button>
<input type="file" id="import-file" accept="application/json" style="display:none">
```

#### 단계 2: JavaScript 구현
내보내기 함수로 메모 데이터를 JSON 파일로 다운로드, 가져오기 함수로 JSON 파일을 읽어 메모 복원.

### 3.4 테스트 계획
#### 단위 테스트 (수동)
1. 내보내기 버튼 클릭 시 JSON 파일이 정상 다운로드되는지 확인
2. 가져오기 버튼 클릭 후 파일 선택 시 메모가 정상 복원되는지 확인
3. 잘못된 파일(포맷 오류) 업로드 시 에러 메시지 표시
4. 기존 메모가 덮어써지는지 확인

#### 엣지 케이스
- 빈 데이터 내보내기/가져오기
- 대량 데이터(수백 개) 처리
- 다른 앱에서 내보낸 파일 호환성

### 3.5 완료 기준
- [ ] 내보내기/가져오기 버튼 UI 구현
- [ ] JSON 파일 다운로드/업로드 정상 동작
- [ ] 데이터 구조 호환성 유지
- [ ] 모든 테스트 케이스 통과

### 3.6 예상 소요 시간
- 구현: 1시간
- 테스트: 20분

### 3.7 다음 단계
구현 완료 후 PDCA 사이클의 Do 단계로 이동하여 실제 코딩 시작.

## 4. 카카오톡 공유 (KakaoTalk Share)

### 4.1 목적
메모 내용을 카카오톡으로 간편하게 공유할 수 있도록 하여, 협업·정보 전달 편의성을 높인다.

### 4.2 구현 개요
- **UI**: 메모 상세/목록에 "카카오톡 공유" 버튼 추가
- **로직**: 카카오톡 JS SDK(링크 공유) 또는 Web Share API 활용
- **공유 방식**: 텍스트(제목+내용) 또는 앱 링크 전송
- **제약**: 카카오 개발자 등록, HTTPS 환경 필요, 모바일 브라우저 권장

### 4.3 구현 단계
#### 단계 1: 카카오 개발자 등록 및 앱 키 발급
- [카카오 개발자](https://developers.kakao.com/)에서 앱 생성
- JavaScript 키 발급 및 도메인 등록(예: localhost, 실제 배포 도메인)

#### 단계 2: SDK 스크립트 추가
카카오 JavaScript SDK 스크립트 태그 추가.

#### 단계 3: 공유 버튼 UI 추가
메모 카드에 카카오톡 공유 버튼 추가.

#### 단계 4: JavaScript 공유 로직
SDK 초기화 및 공유 함수 구현, 이벤트 바인딩.

### 4.4 대안: Web Share API (지원 브라우저 한정)
- 별도 SDK 없이 `navigator.share()`로 텍스트 공유
- 카카오톡 앱이 Web Share를 지원하는 경우에만 동작

### 4.5 테스트 계획
#### 단위 테스트 (수동)
1. 공유 버튼 클릭 시 카카오톡 공유창이 정상적으로 뜨는지 확인
2. 메모 제목/내용이 올바르게 전달되는지 확인
3. 모바일/PC 환경에서 동작 확인
4. 앱 키/도메인 등록 오류 시 예외 처리

#### 엣지 케이스
- 비로그인/비설치 환경 안내
- 긴 메모, 특수문자 포함, 빈 메모 등

### 4.6 완료 기준
- [ ] 카카오톡 공유 버튼 UI 구현
- [ ] SDK 연동 및 정상 동작 확인
- [ ] 테스트 케이스 통과

- 구현: 1~2시간 (카카오 개발자 등록 포함)
- 테스트: 30분

## 5. 스와이프 액션 (Swipe Action)

### 5.1 목적
모바일 UX를 강화하기 위해 메모 카드에 스와이프 제스처를 도입, 빠른 삭제/즐겨찾기 토글을 지원한다.

### 5.2 구현 개요
- **UI**: 메모 카드에서 좌/우로 스와이프 시 액션 노출
- **로직**: 터치 이벤트로 스와이프 감지, 일정 거리 이상 이동 시 액션 실행
- **액션**: 좌로 스와이프 → 삭제, 우로 스와이프 → 즐겨찾기 토글
- **피드백**: 애니메이션 및 액션 완료 시 시각적 효과

### 5.3 구현 단계
#### 단계 1: HTML/CSS 준비
- 메모 카드에 스와이프 가능 구조 적용
- 스와이프 시 배경/아이콘 노출 CSS 추가

#### 단계 2: JavaScript 터치 이벤트 처리 (js/app.js)
- touchstart, touchmove, touchend 이벤트로 제스처 감지
- 일정 임계값(예: 60px) 이상 이동 시 액션 실행
- 액션 후 카드 애니메이션 및 상태 갱신

#### 단계 3: 액션 로직 구현
- 좌로 스와이프: 해당 메모 삭제
- 우로 스와이프: 즐겨찾기 토글
- 데스크톱 환경에서는 미동작(모바일 전용)

### 5.4 테스트 계획
- 모바일 기기에서 스와이프 동작 확인
- 삭제/즐겨찾기 토글 정상 동작
- 실수 방지(임계값, 애니메이션 등) 확인

### 5.5 완료 기준
- [ ] 스와이프 제스처 감지 및 액션 구현
- [ ] UI/애니메이션 정상 동작
- [ ] 모든 테스트 케이스 통과

### 5.6 예상 소요 시간
- 구현: 1.5~2시간
- 테스트: 30분

## 6. 키보드 단축키 (Keyboard Shortcuts)

### 6.1 목적
데스크톱 및 태블릿 환경에서 생산성을 높이기 위해 주요 기능에 키보드 단축키를 추가한다.

### 6.2 구현 개요
- **UI**: 별도 UI 변경 없음, 단축키 안내는 도움말/툴팁에 추가 가능
- **로직**: 주요 기능에 단축키 이벤트 바인딩
- **지원 환경**: 데스크톱/태블릿(모바일 키보드 미지원)

### 6.3 단축키 목록

| 기능         | 단축키      | 설명                |
|--------------|-------------|---------------------|
| 저장         | Ctrl+S      | 편집 뷰에서 메모 저장|
| 뒤로가기     | Esc         | 편집 뷰 → 목록 뷰    |
| 새 메모      | Ctrl+N      | 목록 뷰에서 새 메모  |
| 검색 포커스  | Ctrl+F      | 목록 뷰에서 검색창   |
| 삭제         | Ctrl+D      | 편집 뷰에서 메모 삭제|

### 6.4 구현 단계
#### 단계 1: JavaScript 이벤트 처리 (js/app.js)
- window/document에 keydown 이벤트 리스너 추가
- 현재 뷰(state.currentView)에 따라 동작 분기
- 단축키 입력 시 해당 기능 함수 호출
- 기본 브라우저 동작(e.g. Ctrl+S 저장) 방지

#### 단계 2: 안내/도움말
- 단축키 목록을 도움말/툴팁/설정 화면에 안내(선택사항)

### 6.5 테스트 계획
- 각 단축키 입력 시 기능 정상 동작 확인
- 모바일 환경에서 오작동 없는지 확인
- 브라우저 기본 동작 방지 확인(Ctrl+S 등)

### 6.6 완료 기준
- [ ] 주요 기능 단축키 모두 동작
- [ ] 브라우저 기본 동작 방지
- [ ] 안내/도움말에 단축키 표시(선택)
- [ ] 모든 테스트 케이스 통과

## 7. 자동저장 (Auto Save)

### 7.1 목적
사용자가 메모를 작성하는 동안 실수로 앱을 닫거나 브라우저가 충돌해도 데이터를 잃지 않도록 자동 저장 기능을 추가한다.

### 7.2 구현 개요
- **UI**: 변경 없음 (백그라운드 동작)
- **로직**: 입력 이벤트에 debounce 적용, 300ms 후 자동 저장
- **피드백**: 저장 시도 시 시각적 표시 (선택사항)
- **범위**: 제목과 내용 입력 시에만 적용

### 7.3 구현 단계
#### 단계 1: Debounce 유틸리티 함수
입력 이벤트를 지연시키는 debounce 함수 구현.

#### 단계 2: 입력 이벤트 바인딩
제목과 내용 입력 이벤트에 debounce 적용하여 300ms 후 자동 저장.

#### 단계 3: 저장 표시 (선택사항)
저장 시 사용자에게 피드백 제공:

```javascript
function showSaveIndicator() {
    // 토스트 메시지 또는 상태 표시
    const indicator = document.getElementById('save-indicator');
    if (indicator) {
        indicator.textContent = '저장됨';
        indicator.style.opacity = '1';
        setTimeout(() => {
            indicator.style.opacity = '0';
        }, 1000);
    }
}
```

### 7.4 테스트 계획
#### 단위 테스트 (수동)
1. 메모 입력 시 300ms 후 자동 저장되는지 확인
2. 빠른 입력 변경 시 마지막 입력만 저장되는지 확인
3. 앱 재시작 시 자동 저장된 내용이 유지되는지 확인
4. 저장 표시가 정상 동작하는지 확인
